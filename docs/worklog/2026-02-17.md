## Accomplished

### AI-Human Collaborative Annotation System (the big one)

**Vision and design** (`kb/plans/minard-ai-collaboration.md`): Brainstormed and wrote the design document for Minard as a collaborative code understanding platform. The core insight: static analysis reveals *structural* facts, AI reads and forms *semantic* interpretations, humans contribute *domain knowledge and intent*. The disagreements between AI and human annotations are the most valuable signal — they surface where code structure doesn't match architectural intent, or where implicit knowledge isn't captured in source.

The document defines:
- The workflow: load -> AI reads & annotates -> human reviews -> disagreements surface insights
- Annotation taxonomy: semantic tags, quality signals, architecture labels, relationships, summaries, human context
- The fractal principle: same annotation/review cycle at every level (package, module, declaration, relationship)
- Five implementation phases, from query CLI through mismatch intelligence
- Visual interface implications: badges, intelligence cards, architecture maps, guided tours

**Annotation schema and backend** (commit `1117a45`):
- `annotations` table in DuckDB with target_type/target_id/kind/value/source/confidence/status
- REST API: `GET/POST /api/v2/annotations`, `GET/PATCH /api/v2/annotations/:id`
- Python CLI tool (`minard-annotate`) with list/add/confirm/reject/update subcommands for AI agents to write annotations
- Schema version 3.4, idempotent migration on server boot

**Frontend annotation rendering** (commit `6912787`):
- Replaced post-render querySelector + innerHTML injection with pure Sigil render functions set as `HP.prop innerHTML` during Halogen render — fixes timing bugs where cards appeared empty
- Annotation text split into sentences and rendered as bullet points on signature map cards
- SceneCoordinator lazy-loads annotations per module, passes to signature map

**Annotation lifecycle** (commit `8bb02ab`):
- Confirm/Dispute buttons on signature map cards with status-dependent styling (green border for confirmed, red for rejected, amber for stale)
- PATCH endpoint for status updates with optimistic UI update in SceneCoordinator
- **Content-hash stale detection** in Rust loader: SHA-256 of sorted `kind:name:signature` per module stored in `module_metrics.content_hash`. On reload, compares old vs new hashes and marks annotations as "stale" when a module's API surface changes. Forward-compatible schema migration.
- **Codebase report endpoint** (`GET /api/v2/report`): markdown report of all annotations grouped by package/module/kind, with status icons and module stats

## Explored But Not Pursued

- **Purity as a persistent color mode**: The peek pattern is better — purity is a quick question, not a sustained analysis. A persistent mode would compete with existing color modes.
- **Counting all declarations for purity**: Only value declarations have meaningful return types. Type synonyms and data types are neither pure nor effectful.
- **Annotation filtering/sorting by staleness**: The stale detection pipeline works but the UI doesn't yet have dedicated filter controls.

## Decisions Made

- **Annotations as a first-class data layer**: Not just metadata — the annotation system is Minard's pathway from pure visualization to collaborative understanding. The schema is deliberately rich enough to support multi-source (AI, human, static analysis) annotations with confidence tracking and disagreement surfacing.
- **Status lifecycle: proposed -> confirmed/rejected, with stale**: Content hashing detects when source changes invalidate annotations. This means AI annotations survive loader runs but get flagged when the code they describe changes.
- **Text target IDs**: Annotations use module/declaration *names* as target_id (not integer PKs) so they survive data reloads where IDs change.
- **Peek, not mode** for purity: Hold-key overlay keeps the color mode menu clean.
- **Blue/amber palette for purity**: Consistent across arc diagram labels and treemap overlay.
- **Exported API surface only**: Purity peek at package level shows the API character, not internals.
