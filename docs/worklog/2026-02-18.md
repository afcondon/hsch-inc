## Decisions Made

- **Additive, not replacement**: Each reply is a new annotation. `supersedes` = "responds to", not "replaces". The full chain is preserved. This is the key design choice — disagreements are the most valuable artifact.
- **No auto-status-change on reply**: Replying doesn't mark the parent as stale or superseded. Confirm/Dispute remain separate quick gestures. The chain is purely additive.
- **Collapse is UI-only**: Thread collapse stored in component state (`Set Int`), not persisted. All annotations always exist in the database. The report always shows full threads.
- **Reply on any status**: You can reply to confirmed, rejected, proposed, or stale annotations. The conversation never closes.
- **Skill uses curl, not special tooling**: Any Claude session can annotate. The skill documents the protocol; the API does the work.
- **Thread display**: Root annotation primary, replies indented with 2px left border and 0.8 opacity, source labels on each entry. Collapsed view shows only the latest entry.

## Explored But Not Pursued

- **Persisting collapse state**: Could store collapsed thread IDs in localStorage or a user preferences table. Decided against — collapse is a transient reading preference, not durable state.
- **Auto-threading by kind**: Considered automatically grouping annotations of the same kind into a thread even without `supersedes`. Rejected — explicit threading via `supersedes` is clearer and preserves independent observations.
