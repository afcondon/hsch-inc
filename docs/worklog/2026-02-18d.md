## Decisions Made

- **`execSync` blocks intentionally**: The loader writes to DuckDB while the Node event loop is blocked, avoiding any concurrent access issues. The server is unresponsive for 3-15s during a load — acceptable for a local dev tool
- **Boot-time project detection**: AppShell checks for projects before loading any codebase data. Empty DB → straight to ProjectManagement with no wasted API calls
- **Effect typing for FFI**: `validatePathJson` and `runLoaderSync` are `Effect String` (not pure) since they perform filesystem reads and `execSync`. JS side uses `() =>` thunk wrapper
- **`for_` over cascade delete**: Replaced initial complex `execAll` with `unsafeHead/unsafeTail` with clean `Data.Traversable.for_` loop
- **Loader binary resolution**: Defaults to `./minard-loader/target/release/minard-loader` relative to server cwd, overridable via `MINARD_LOADER_PATH` env var

## Explored But Not Pursued

- **WebSocket progress streaming during load**: Considered streaming loader output to the browser. `execSync` makes this impossible by design — and the 3-15s blocking window is short enough that a pulsing indicator suffices
- **Project scoping for all queries**: The API accepts `?project=N` on packages/modules endpoints but the frontend doesn't yet use it for filtering. Left for a future multi-project session

## Parking Lot

- **Project switching**: Currently loading a project triggers a full data reload. Could be smarter about caching per-project data
- **Empty state testing**: Need to test with a genuinely empty database to verify the welcome flow end-to-end
- **Validation UX**: Could add real-time validation as the user types (debounced), rather than requiring a button click
- **Project labels/tags**: The `loadProject` body accepts `name` and `label` overrides but the UI doesn't expose them yet
